# Frontend Slides

创建完全在浏览器中运行的零依赖、动画丰富的 HTML 演示文稿。

## 核心原则

1. **零依赖** — 单个 HTML 文件，内联 CSS/JS。无需 npm，无需构建工具。
2. **展示，而非讲述** — 生成视觉预览，而非抽象的选择题。让用户通过看到的效果来发现自己的喜好。
3. **独特设计** — 拒绝千篇一律的"AI 敷衍作品"。每个演示文稿都必须感觉是精心定制。
4. **视口适配（强制要求）** — 每张幻灯片必须精确适配 100vh。幻灯片内绝不滚动。内容溢出？拆分为多张幻灯片。

## 设计美学

你往往趋向于输出通用的、"随大流"的结果。在前端设计中，这会产生用户所谓的"AI 敷衍作品"美学。要避免这一点：做出有创意、独特、令人惊喜的前端。

重点关注：

- **字体排版**：选择美观、独特、有趣的字体。避免 Arial 和 Inter 等通用字体；选择能提升前端美学品质的独特选择。
- **颜色与主题**：坚持一致的美学风格。使用 CSS 变量保持一致性。主色调配以锐利的点缀色，优于平淡、均匀分布的配色方案。从 IDE 主题和文化美学中汲取灵感。
- **动效**：使用动画实现效果和微交互。HTML 优先使用纯 CSS 解决方案。React 中有 Motion 库时使用它。专注于高impact时刻：一次精心编排的页面加载配合交错显示的动画（animation-delay），比零散的微交互更能带来愉悦。
- **背景**：创造氛围和深度，而非默认使用纯色。叠加 CSS 渐变，使用几何图案，或添加与整体美学相符的上下文效果。

避免通用的 AI 生成美学：

- 过度使用的字体族（Inter、Roboto、Arial、系统字体作为标题）
- 俗套的配色方案（尤其是白底紫色渐变）
- 可预测的布局和组件模式
- 缺乏上下文特定个性的模板化设计

创意解读，做出感觉真正为上下文量身定制的意外选择。在浅色和深色主题、不同字体、不同美学之间变化。你仍然倾向于在多次生成中收敛于常见选择（例如 Space Grotesk）。避免这一点：关键是你要跳出思维定式！

## 视口适配规则

这些不变规则适用于**每个**演示文稿中的**每张**幻灯片：

- 每个 `.slide` 必须有 `height: 100vh; height: 100dvh; overflow: hidden;`
- **所有**字体大小和间距必须使用 `clamp(min, preferred, max)` — 绝不使用固定 px/rem
- 内容容器需要 `max-height` 约束
- 图片：`max-height: min(50vh, 400px)`
- 高度断点要求：700px、600px、500px
- 包含 `prefers-reduced-motion` 支持
- 绝不要直接否定 CSS 函数（`-clamp()`、`-min()`、`-max()` 会被浏览器静默忽略）— 使用 `calc(-1 * clamp(...))` 代替

**生成时，读取 `viewport-base.css` 并将其完整内容包含在每个演示文稿中。**

### 每张幻灯片的内容密度限制

| 幻灯片类型 | 最大内容 |
| --------- | -------- |
| 标题幻灯片 | 1 个标题 + 1 个副标题 + 可选的标语 |
| 内容幻灯片 | 1 个标题 + 4-6 个要点 **或** 1 个标题 + 2 段文字 |
| 功能网格 | 1 个标题 + 最多 6 张卡片（2x3 或 3x2）|
| 代码幻灯片 | 1 个标题 + 8-10 行代码 |
| 引言幻灯片 | 1 条引言（最多 3 行）+ 归属 |
| 图片幻灯片 | 1 个标题 + 1 张图片（最大 60vh 高度）|

**内容超出限制？拆分为多张幻灯片。绝不拥挤，绝不滚动。**

## 阶段 0：检测模式

确定用户想要什么：

- **模式 A：新演示文稿** — 从头开始创建。跳转到阶段 1。
- **模式 B：PPT 转换** — 转换 .pptx 文件。跳转到阶段 4。
- **模式 C：增强** — 改进现有的 HTML 演示文稿。读取它、理解它、增强它。**遵循下面的模式 C 修改规则。**

### 模式 C：修改规则

增强现有演示文稿时，视口适配是最大风险：

1. **添加内容前：** 计算现有元素，对照密度限制检查
2. **添加图片：** 必须有 `max-height: min(50vh, 400px)`。如果幻灯片已有最大内容，拆分为两张幻灯片
3. **添加文字：** 每张幻灯片最多 4-6 个要点。超出限制？拆分为续幻灯片
4. **任何修改后验证：** `.slide` 有 `overflow: hidden`，新元素使用 `clamp()`，图片有视口相关的 max-height，内容在 1280x720 下适配
5. **主动重组：** 如果修改会导致溢出，自动拆分内容并告知用户。不要等到被要求

**向现有幻灯片添加图片时：** 将图片移到新幻灯片，或先减少其他内容。绝不要在未检查现有内容是否已占满视口的情况下添加图片。

## 阶段 1：内容发现（新演示文稿）

**在单个 AskUserQuestion 调用中提出所有问题**，让用户一次性填写：

**问题 1 — 用途**（header: "Purpose"）：
这个演示文稿的用途是什么？选项：融资推介 / 教学教程 / 会议演讲 / 内部汇报

**问题 2 — 长度**（header: "Length"）：
大约需要多少张幻灯片？选项：短 5-10 / 中等 10-20 / 长 20+

**问题 3 — 内容**（header: "Content"）：
你的内容准备好了吗？选项：所有内容已准备好 / 粗略笔记 / 只有主题

**问题 4 — 内嵌编辑**（header: "Editing"）：
需要在生成后在浏览器中直接编辑文字吗？选项：

- "是（推荐）" — 可以在浏览器中编辑，自动保存到 localStorage，导出文件
- "否" — 仅演示用，文件更小

**记住用户的编辑选择 — 这决定了阶段 3 是否包含编辑相关代码。**

如果用户有内容，要求他们分享。

### 步骤 1.2：图片评估（如果提供了图片）

如果用户选择"无图片" → 跳转到阶段 2。

如果用户提供图片文件夹：

1. **扫描** — 列出所有图片文件（.png、.jpg、.svg、.webp 等）
2. **查看每张图片** — 使用 Read 工具（Claude 是多模态的）
3. **评估** — 对于每张：显示什么，是否可用（附原因），代表什么概念，主导颜色
4. **共同设计大纲** — 精选图片与文字一起决定幻灯片结构。这不是"先规划幻灯片再添加图片"— 从一开始就围绕两者设计（例如 3 张截图 → 3 张功能幻灯片，1 个 logo → 标题/结尾幻灯片）
5. **通过 AskUserQuestion 确认**（header: "Outline"）："这个幻灯片大纲和图片选择看起来对吗？"选项：看起来不错 / 调整图片 / 调整大纲

**预览中的 Logo：** 如果发现了可用的 logo，将其嵌入（base64）到阶段 2 的每个风格预览中 — 用户可以看到其品牌的三种不同风格呈现。

## 阶段 2：风格发现

**这是"展示，而非讲述"的阶段。** 大多数人无法用语言表达设计偏好。

### 步骤 2.0：风格路径

询问他们想如何选择（header: "Style"）：

- "给我看选项"（推荐）— 基于氛围生成 3 个预览
- "我知道我想要什么" — 直接从预设列表中选择

**如果直接选择：** 显示预设选择器并跳转到阶段 3。可用预设定义在 [STYLE_PRESETS.md](STYLE_PRESETS.md)。

### 步骤 2.1：氛围选择（引导式发现）

询问（header: "Vibe"，multiSelect: true，最多 2 个）：
观众应该有怎样的感受？选项：

- Impressed/Confident（印象深刻/自信）— 专业、值得信赖
- Excited/Energized（兴奋/充满能量）— 创新、大胆
- Calm/Focused（冷静/专注）— 清晰、思考性
- Inspired/Moved（受启发/感动）— 情感、难忘

### 步骤 2.2：生成 3 个风格预览

基于氛围，生成 3 个不同的单幻灯片 HTML 预览，展示字体排版、颜色、动画和整体美学。阅读 [STYLE_PRESETS.md](STYLE_PRESETS.md) 了解可用预设及其规格。

| 氛围 | 建议预设 |
| ---- | -------- |
| Impressed/Confident | Bold Signal, Electric Studio, Dark Botanical |
| Excited/Energized | Creative Voltage, Neon Cyber, Split Pastel |
| Calm/Focused | Notebook Tabs, Paper & Ink, Swiss Modern |
| Inspired/Moved | Dark Botanical, Vintage Editorial, Pastel Geometry |

将预览保存到 `.claude-design/slide-previews/`（style-a.html、style-b.html、style-c.html）。每个都应自包含，约 50-100 行，展示一张动画标题幻灯片。

自动为用户打开每个预览。

### 步骤 2.3：用户选择

询问（header: "Style"）：
你喜欢哪个风格预览？选项：Style A: [名称] / Style B: [名称] / Style C: [名称] / 混合元素

如果选择"混合元素"，询问具体细节。

## 阶段 3：生成演示文稿

使用阶段 1 的内容（文字，或文字 + 精选图片）和阶段 2 的风格生成完整演示文稿。

如果提供了图片，幻灯片大纲已从步骤 1.2 结合了图片。如果没有，CSS 生成的视觉效果（渐变、形状、图案）提供视觉兴趣 — 这是一条完全支持的一等路径。

**生成前，阅读这些支持文件：**

- [html-template.md](html-template.md) — HTML 架构和 JS 功能
- [viewport-base.css](viewport-base.css) — 强制 CSS（完整包含）
- [animation-patterns.md](animation-patterns.md) — 所选氛围的动画参考

**关键要求：**

- 单个自包含 HTML 文件，所有 CSS/JS 内联
- 在 `<style>` 块中包含 viewport-base.css 的**完整**内容
- 使用 Fontshare 或 Google Fonts 的字体 — 绝不使用系统字体
- 添加解释每个部分的详细注释
- 每个部分需要一个清晰的 `/* === 部分名称 === */` 注释块

## 阶段 4：PPT 转换

转换 PowerPoint 文件时：

1. **提取内容** — 运行 `python scripts/extract-pptx.py <input.pptx> <output_dir>`（如需安装 python-pptx：`pip install python-pptx`）
2. **与用户确认** — 呈现提取的幻灯片标题、内容摘要和图片数量
3. **风格选择** — 继续阶段 2 进行风格发现
4. **生成 HTML** — 转换为所选风格，保留所有文字、图片（来自 assets/）、幻灯片顺序和演讲者备注（作为 HTML 注释）

## 阶段 5：交付

1. **清理** — 删除 `.claude-design/slide-previews/`（如果存在）
2. **打开** — 使用 `open [filename].html` 在浏览器中启动
3. **总结** — 告诉用户：
   - 文件位置、风格名称、幻灯片数量
   - 导航：方向键、空格键、滚动/滑动、点击导航点
   - 如何自定义：`:root` CSS 变量用于颜色，`font link` 用于字体排版，`.reveal` 类用于动画
   - 如果启用了内嵌编辑：悬停在左上角或按 E 进入编辑模式，点击任何文字进行编辑，Ctrl+S 保存

## 阶段 6：分享与导出（可选）

交付后，**询问用户：**_"你想分享这个演示文稿吗？我可以将其部署到在线 URL（在任何设备包括手机上都能运行）或导出为 PDF。"_

选项：

- **部署到 URL** — 可分享的链接，在任何设备上都能用
- **导出为 PDF** — 适用于邮件、Slack、打印的通用文件
- **两者都要**
- **不用了谢谢**

如果用户拒绝，在此停止。如果选择一个或两者，继续下面步骤。

### 6A：部署到在线 URL（Vercel）

这会将演示文稿部署到 Vercel — 一个免费托管平台。链接在任何设备（手机、平板、笔记本电脑）上都能用，并且会一直保留，直到用户将其删除。

**如果用户从未部署过，请逐步引导：**

1. **检查 Vercel CLI 是否已安装** — 运行 `npx vercel --version`。如果找不到，先安装 Node.js（macOS 上 `brew install node`，或从 https://nodejs.org 下载）。

2. **检查用户是否已登录** — 运行 `npx vercel whoami`。
   - 如果**未登录**，解释：_"Vercel 是一个免费托管服务。你需要一个账户来部署。让我引导你完成："_
     - 步骤 1：要求用户在其浏览器中访问 https://vercel.com/signup
     - 步骤 2：他们可以用 GitHub、Google、邮箱注册 — 什么最方便就用什么
     - 步骤 3：注册后，运行 `vercel login` 并按照提示操作（它会打开浏览器窗口进行授权）
     - 步骤 4：用 `vercel whoami` 确认登录
   - 等待用户确认他们已登录后再继续。

3. **部署** — 运行部署脚本：

   ```bash
   bash scripts/deploy.sh <path-to-presentation>
   ```

   脚本接受文件夹（带 index.html）或单个 HTML 文件。

4. **分享 URL** — 告诉用户：
   - 在线 URL（来自脚本输出）
   - 它在任何设备上都能用 — 他们可以短信、Slack 或邮件发送
   - 稍后删除：访问 https://vercel.com/dashboard 并删除项目
   - Vercel 免费额度很慷慨 — 不会被收费

**⚠ 部署注意事项：**

- **本地图片/视频必须随 HTML 一起传输。** 部署脚本自动检测 HTML 中通过 `src="..."` 引用的文件。但如果演示文稿通过 CSS `background-image` 或异常路径引用文件，这些可能会被遗漏。**部署前验证：** 打开部署的 URL 检查所有图片是否加载。如果有损坏，最安全的修复是将 HTML 和所有资产放入单个文件夹，然后部署文件夹而非单个 HTML 文件。
- **当演示文稿有很多资产时，优先使用文件夹部署。** 如果演示文稿位于带有图片的文件夹中（例如 `my-deck/index.html` + `my-deck/logo.png`），直接部署文件夹：`bash scripts/deploy.sh ./my-deck/`。这比部署单个 HTML 文件更可靠，因为整个文件夹内容原样上传。
- **带空白的文件名可以工作但可能有问题。** 脚本处理文件名中的空格，但 Vercel URL 将空格编码为 `%20`。如果可能，避免图片文件名中使用空格。如果用户的图片有空格，脚本会处理 — 但如果图片仍然损坏，重命名文件使用连字符代替空格是修复方法。
- **重新部署会更新相同的 URL。** 再次对同一演示文稿运行部署脚本会覆盖之前的部署。URL 保持不变 — 无需分享新链接。

### 6B：导出为 PDF

这会将每张幻灯片截图并合并为 PDF。适用于邮件附件、嵌入文档或打印。

**注意：** 动画和交互性不会被保留 — PDF 是静态快照。这是正常的和预期的；告诉用户以免他们感到惊讶。

1. **运行导出脚本：**

   ```bash
   bash scripts/export-pdf.sh <path-to-html> [output.pdf]
   ```

   如果未给出输出路径，PDF 会保存在 HTML 文件旁边。

2. **幕后发生了什么**（向用户简要解释）：
   - 无头浏览器以 1920×1080（标准宽屏）打开演示文稿
   - 逐张截图每张幻灯片
   - 所有截图合并为单个 PDF
   - 脚本需要 Playwright（浏览器自动化工具）— 如果缺失会自动安装

3. **如果 Playwright 安装失败：**
   - 最常见的问题是 Chromium 未下载。运行：`npx playwright install chromium`
   - 如果仍然失败，可能是网络/防火墙问题。要求用户在不同的网络上尝试。

4. **交付 PDF** — 脚本自动打开它。告诉用户：
   - 文件位置和大小
   - 它在任何地方都能用 — 邮件、Slack、Notion、Google Docs、打印
   - 动画被替换为最终视觉状态（仍然看起来很棒，只是静态的）

**⚠ PDF 导出注意事项：**

- **首次运行较慢。** 脚本安装 Playwright 并将 Chromium 浏览器（~150MB）下载到临时目录。每次运行都会发生一次。警告用户第一次可能需要 30-60 秒 — 同一会话中的后续导出会更快。
- **幻灯片必须使用 `class="slide"`。** 导出脚本通过查询 `.slide` 元素来查找幻灯片。如果演示文稿使用不同的类名，脚本会报告"找到 0 张幻灯片"并失败。本技能生成的所有演示文稿都使用 `.slide`，所以这仅对外部创建的 HTML 有影响。
- **本地图片必须可通过 HTTP 加载。** 脚本启动本地服务器并通过它加载 HTML（这样 Google Fonts 和相对图片路径才能工作）。如果图片使用绝对文件系统路径（例如 `src="/Users/name/photo.png"`）而不是相对路径（例如 `src="photo.png"`），它们将无法加载。生成的演示文稿始终使用相对路径，但转换的或用户提供的演示文稿可能不会 — 检查并在需要时修复。
- **本地图片会出现在 PDF 中**，只要它们位于与 HTML 文件相同的目录中（或相对于 HTML 文件）。导出脚本通过 HTTP 提供 HTML 的父目录，因此像 `src="photo.png"` 这样的相对路径可以正确解析 — 包括带空格的文件名。如果图片仍然不出现，检查：(1) 图片文件实际存在于引用路径，(2) 路径是相对的，不是绝对文件系统路径如 `/Users/name/photo.png`。
- **大型演示文稿产生大型 PDF。** 每张幻灯片被捕获为完整的 1920×1080 PNG 截图。18 张幻灯片的演示文稿可以产生约 20MB 的 PDF。如果 PDF 超过 10MB，询问用户：_"PDF 大小为 [大小]。你想让我压缩它吗？它会看起来稍微不那么清晰，但文件会小很多。"_ 如果是，重新运行导出并使用 `--compact` 标志：
  ```bash
  bash scripts/export-pdf.sh <path-to-html> [output.pdf] --compact
  ```
  这会以 1280×720 而非 1920×1080 渲染，通常可以减少 50-70% 的文件大小，同时视觉差异最小。
